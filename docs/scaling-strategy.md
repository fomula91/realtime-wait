# Scaling Strategy

이 문서는 데모용 구조(현재)와 실제 운영용 구조(확장안)를 분리해 정리한다. 데모의 단순함을 유지하면서, 운영 전환 시의 방향을 명확히 남기는 것이 목적이다.

## 현재(데모) 구조

```text
Browser ──polling──▶ Cloudflare Worker ──▶ D1
```

- 실시간성: polling (참가자 5초 / 운영자 3초).
- 장점: 요청량 예측 용이, 무료 한도 운영, 서버리스로 운영 부담 최소.
- 한계: 3~5초 지연, 참가자 수 증가 시 폴링 요청 선형 증가, D1 쓰기 동시성.

## 부하 증가 시 단계적 대응

### 1단계 — 폴링 최적화 (구조 변경 없음)

- 상태 변화가 없으면 `304`/ETag 로 응답 본문 절약.
- 폴링 주기 동적 조정(대기 순번이 멀면 주기 ↑).
- 대기열 조회 결과를 짧은 TTL 로 캐싱(Workers Cache / Runtime Cache).

### 2단계 — 실시간 전송으로 전환 (SSE)

```text
Browser ──SSE──▶ Realtime Server ──sub──▶ Redis Pub/Sub
                                   ▲
Worker(API) ──publish(상태변경)────┘
```

- 상태 변경(호출/체크인 등) 시점에만 push → 폴링 요청 제거.
- SSE 는 단방향이라 WebSocket 보다 운영이 단순(대기열 알림 용도에 충분).
- 장기 연결 관리가 필요하므로 전용 서버(분리)가 필요 → ADR-0005.

### 3단계 — 상태 저장소 확장

- D1 → MySQL/Postgres 이전 검토(쓰기 동시성·트랜잭션·인덱스).
- 대기열 카운터 등 hot path 는 Redis 로 분리.
- 읽기 부하는 read replica / 캐시 계층으로 흡수.

### 4단계 — 실시간 서버 스택 선택

- Go 또는 NestJS 기반 realtime server 검토(연결 수·메모리 효율).
- 수평 확장 시 Redis Pub/Sub 로 인스턴스 간 메시지 팬아웃.

## 데모 ↔ 운영 분리 원칙 (ADR-0006)

- 데모는 비용 0·운영 부담 0 을 최우선으로 유지한다.
- 운영 전환은 트래픽 근거가 생겼을 때 단계적으로 적용한다.
- shared 패키지의 타입/스키마는 두 구조에서 공통으로 재사용한다.

## 전환 트리거(예시)

| 신호 | 대응 단계 |
| --- | --- |
| 폴링 요청이 무료 한도의 70% 도달 | 1단계 (폴링 최적화) |
| 체감 지연/즉시성 불만 | 2단계 (SSE) |
| D1 쓰기 경합/지연 증가 | 3단계 (저장소 확장) |
| 동시 연결 수 급증 | 4단계 (전용 realtime 서버) |
