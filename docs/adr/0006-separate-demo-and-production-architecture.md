# ADR-0006: 데모 구조와 운영 구조의 분리

- 상태: 채택
- 일자: 2026-06-23

## 맥락

하나의 구조로 데모와 운영을 모두 만족시키려 하면, 데모가 과도하게 복잡해지거나 운영 요구를 충족하지 못한다.

## 결정

데모 구조와 운영 구조를 **명시적으로 분리**해 문서화하고, 데모는 단순함·무비용을 우선한다.

| 구분 | 데모(현재) | 운영(확장안) |
| --- | --- | --- |
| 실시간 | polling | SSE + Redis Pub/Sub (ADR-0005) |
| 컴퓨트 | Cloudflare Workers | 전용 realtime 서버(Go/NestJS) + API |
| 저장소 | D1 | MySQL/Postgres + Redis |

## 근거

- 데모는 포트폴리오에서 "동작하는 것"과 "판단 근거"를 보여주는 게 목적.
- 운영 전환은 트래픽 근거가 생겼을 때 단계적으로(scaling-strategy 트리거).
- shared 타입/스키마·API 형식을 공통 자산으로 두어 전환 비용을 낮춤.

## 트레이드오프

- 두 구조를 모두 문서로 유지해야 함(문서 관리 비용).
- 데모 코드가 운영 구조를 그대로 담지는 않음 → 의도적 선택임을 README/ADR 에 명시.

## 결과

- 데모: 이 저장소의 실제 구현.
- 운영: `scaling-strategy.md` 의 단계적 전환 + ADR-0005.
