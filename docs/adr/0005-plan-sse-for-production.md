# ADR-0005: 운영 단계 실시간 전송으로 SSE 계획

- 상태: 제안(운영 전환 시)
- 일자: 2026-06-23

## 맥락

데모는 polling 을 쓰지만(ADR-0004), 트래픽이 늘고 즉시성 요구가 커지면 폴링 요청량이 부담이 된다.

## 결정(방향)

운영 단계에서는 SSE 기반 push 로 전환한다. 상태 변경 시점에만 클라이언트에 전송한다.

```text
Worker(API) ──publish──▶ Redis Pub/Sub ──▶ Realtime Server ──SSE──▶ Browser
```

## 근거

- 대기열 알림은 서버→클라이언트 단방향이라 SSE 로 충분(WebSocket 불필요).
- 상태 변경 시에만 전송 → 폴링의 선형 요청 증가 문제 해소.
- Redis Pub/Sub 로 다중 인스턴스 팬아웃이 단순.

## 트레이드오프

- 장기 연결 관리가 필요 → 서버리스 대신 전용 realtime 서버 필요.
- 연결 상태/재연결/백프레셔 등 운영 복잡도 증가.

## 비고

- 전용 서버 스택(Go/NestJS)·저장소 확장은 scaling-strategy 및 ADR-0006 참고.
- 데모 코드와 충돌 없이 도입할 수 있도록 API 응답 형식·shared 타입을 재사용한다.
